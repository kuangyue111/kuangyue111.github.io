BMS故障诊断：两态状态机+滤波延迟的可靠故障检测

前言

电池管理系统（BMS）在储能、电动汽车、UPS等高压直流应用中承担着核心的安全守卫职责。一个错误的故障判断——无论是漏报还是误 报——都可能带来严重后果：漏报导致过充过放损毁电芯甚至起火，误报导致系统频繁保护停机、影响生产和用户体验。

如何在噪声环境下做到”不该报的不报，该报的必须报”？本文基于真实固件代码，深入剖析其故障诊断体系的设计思路与实现细节。

一、两态故障状态机：简单却不简陋

1.1 状态定义

打开 DiagInterface.h，最先映入眼帘的是故障滤波状态的枚举定义：

看起来只有四个值，实际上业务层只会看到 eErr_A 和 eErr_B 两个稳定态，eErr_AtoB 和 eErr_BtoA是滤波器的内部过渡态，上层不应直接使用，这是一种”对外两态、内部四态”的封装策略。

1.2滤波状态结构体

每路故障都维护一个独立的 t_FilState 实例。nowState 由各路检测任务实时写入，filState才是系统真正判断”有没有故障”的依据。

1.3滤波参数配置

AtoBTime 是”故障确认时间”，物理含义是：传感器连续检测到异常超过 AtoBTime 个 tick，才认定故障成立。BtoATime是”恢复确认时间”，防止故障在边界值附近反复横跳。滤波任务 DiagErrFilterTask() 的调用周期在 DiagFilter.h 中明确注释：

即每50ms 推进一次滤波计数。如果 AtoBTime 设为 6，则需要 300ms持续异常才会真正触发保护——这对抑制传感器采样毛刺非常有效。

1.4 为什么要分离nowState 与 filState？

在实际工程中有一类常见的陷阱：传感器在采样边界（阈值附近）时，受温度漂移、接触电阻变化等影响，读数在阈值上下来回跳变。如果直接用即时值判断，每次跳变都会触发保护/恢复动作，系统频繁振荡。

滤波器的本质是对时间轴的”多数表决”：只有信号在足够长的时间窗口内持续保持异常，才认为这是真正的故障，而不是噪声。这正是两态状态机+时间滤波的核心价值所在。

例如 eERR_ID_CEL_VH1（单体过压一级保护）的 errSort 对应 eProSort_ChgPro，触发时切断充电继电器；而eERR_ID_CEL_VH2（二级预警）对应 eProSort_ChgLim，只限制充电电流，不直接断路。这形成了”先限流-后断路”的渐进式保护策略，在保证安全的同时减少不必要的强制停机。

三、通信看门狗机制：从控节点的心跳监测

3.1 设计背景

在多从控BSU（Battery Slave Unit）架构中，主控 BMU 通过内部 CAN 总线与最多 32 个从控周期性通信（DIAG_SLAVE_NUM = 32，DiagUser.h:25）。如果某个从控节点因MCU 故障、CAN收发器损坏或线束断路而沉默，主控必须及时检测出来，否则继续用失效节点的最后一帧数据做保护判断会产生严重的安全隐患。

3.2 实现机制

核心 API 在 DiagInterface.h:311 处定义：

机制逻辑如下：

  1. 每个从控对应一个独立计数器，DiagCommuCheckTask() 每秒执行一次，执行时先检查计数，然后复位。
  2. 每收到该从控的 CAN 帧，DiagRcvDevMsgAddAPI(addr) 被调用，计数递增。
  3. 如果在 1秒周期内，某从控的计数低于预设帧数阈值 N，判定该节点通信故障，触发 eERR_ID_INT_COFF 或 eERR_ID_CELL_MISS。

4.2 为什么用位域而不是枚举？

LTC6813 每片管理 18 路单体电压，一个从控 PCB 可能级联多片。用u32 offCell 位图来表示哪些通道断线，相比枚举数组：

  • 报文效率高：4 字节传输 32 路通道状态，CAN 帧无需扩展帧。
  • 批量判断快：if (offCell != 0) 一次操作即可判断该从控是否有任何通道异常。
  • 按位查询精准：(offCell >> cellNum) & 1 取出特定通道状态，无需遍历。

五、继电器粘连检测：高压安全的最后一道门

5.1 粘连的危险性

继电器（接触器）粘连是指触点在断开指令后仍保持导通。在高压电池系统中，粘连的主继电器意味着：

  • 充电停止指令无效，电芯持续过充。
  • 系统”断电”但高压母线仍然带电，危及维修人员。

因此，粘连检测不是”有最好”，而是安全规范的强制要求（如 ISO 26262 ASIL-B 级功能安全）。

六、绝缘监测（接地故障检测）：高压电气安全的基础

6.1 为什么需要绝缘监测

高压电池包采用浮地（Floating Ground）拓扑，正负极均不接车身地/大地。这种设计下，单点绝缘失效（例如电缆破皮接地）不会立即形成回路，但如果随后发生第二点绝缘失效，就会产生短路电流，极有可能引发触电或火灾。绝缘监测（Insulation Monitoring Device，IMD）的作用是在第一点失效时就发出警告，避免双点失效。

6.2 测量原理

七、诊断结果对SOC 算法的保护：闭环安全设计

7.1 采样异常时冻结数据更新

SOC（State of Charge）计算依赖电压和电流的持续积分。一旦采样链路异常，若仍用错误数据更新 SOC，会导致积分漂移——最终 SOC显示值与实际荷电量严重偏离，造成续航里程估算错误或保护阈值失效。

DiagInterface.h:773 中定义了核心保护 Hook：

SOC 计算模块在更新内部状态前会调用这些 Hook：若返回异常，则跳过本轮电压修正（OCV修正）和温度修正，维持上一拍的估算值，避免垃圾数据污染积分器。

7.2 通信故障时停止均衡

均衡（Balancing）通过微小电流将高电量单体的能量耗散或转移，需要持续读取各单体电压来做决策。通信故障时，主控无法获得从控的实时电压，如果此时仍下发均衡控制指令，可能产生以下问题：

  • 均衡目标已恢复正常但指令仍在执行 → 过均衡，降低容量
  • 从控侧均衡 MOSFET 已通但无监督→ 热失控风险

因此当eERR_ID_INT_COFF 或 eERR_ID_CELL_MISS 触发时，均衡控制器会强制停止所有均衡通道，直到通信恢复并重新建链。

7.3 过压/欠压时强制修正 SOC

单体过压（eERR_ID_CEL_VH1）说明电池实际上已经充满，但 SOC 积分值可能因效率误差而小于 100%。此时系统会将 SOC强制修正到目标上限（STEP_CORR_FIN_HSOC = 988，即 98.8%），消除积分误差的累积。

类似地，欠压保护触发时，SOC 下限被修正到接近 0（STEP_CORR_FIN_LSOC = 13，即 1.3%），为下次充电提供准确的起始点。

这是一种典型的”用物理约束修正估算偏差”的工程策略——把物理事件（满充截止电压到达）当作 SOC的标定触发器，抵消长期积分产生的漂移。

八、整体架构回顾：各模块如何串联

这条数据流并非单向通路，而是一个闭环体系：SOC计算产生的偏差信号（例如满充/满放触发）反过来修正电量估算，而诊断模块则在数据链路有问题时随时”拔掉”可能污染 SOC的数据源。这正是真正意义上的”诊断与估算协同”，而非两个独立子系统各行其是。

这六类任务与 DiagErrFilterTask()（50ms 调度）分属不同频率：Maker 负责”写入实时判断结果”，Filter负责”对这些结果做时间积分”。两者解耦的好处在于：不同物理量的采样率不同（电流 10ms 一次，温度 100ms 一次），Maker可以按各自的合理频率更新nowState，而 Filter 统一以固定节拍推进滤波计数器，不会因为某路信号更新频率低而漏判。

九、故障历史记录机制：事后溯源的工程基础

9.1 记录分类

每个故障事件在”产生”和”消除”时各写一条记录，而不仅仅是产生时写入。这使得维护人员可以从历史记录中直接读出某次过压保护的持续时长（消除时间戳 - 产生时间戳），而不是仅仅知道”它发生过”。

历史记录容量支持最多 30 天回溯（DiagChangHisErrDisplayAPI 的 date 参数范围 0–30），每条记录结构体长度为 6个Word（ERR_RCD_ERRINFO_LEN = 6），包含故障代码前缀、故障编号、附加诊断信息（如发生故障时的电压值或温度值）。

十、SOS 安全等级综合评估：从单点故障到整体风险

10.1 设计动机

单路故障的严重程度是相对孤立的：一节单体轻微过压和一节单体严重过压，保护动作不同；而如果同时存在过压、高温和通信故障，组合风险远高于三者简单叠加。SOS（Safety of System）机制就是用来做这种”系统级风险综合评估”的。

10.2 触发原因枚举

DiagSos.h 中列出了所有 SOS 触发原因（eSos_Reasion）：

SOS 任务以 100ms 周期运行（DiagGroupSosTask()），汇聚所有当前激活的故障原因，计算出一个综合安全等级（DiagGetGroupSosLevel()），由上层状态机据此决策是否进入紧急停止、降功率运行或正常停止。

10.3 与普通故障 ID 的区别

普通故障 ID（e_ERR_ID）是一对一的：某路传感器超阈值 → 某个具体故障。SOS 评估是多对一的：多个故障原因共同输入，输出一个”系统整体处于几级危险”的结论。这两层抽象都不可或缺——前者用于精确定位问题根源，后者用于决定系统级响应烈度。

十一、实际开发中的几点工程经验

以上代码分析揭示了几个在BMS 固件开发中值得重点关注的设计模式：

  1. 滤波参数调校是最容易踩坑的地方

AtoBTime 设太短：噪声引发误保护，用户投诉频繁断路。设太长：真实故障响应滞后，可能损伤电芯。实际项目中，这个参数通常需要在台架测试中结合具体传感器噪声特性和保护响应时间要求联合标定，不能直接照搬其他平台经验值。

  1. 位域记录是大规模节点系统的标配

当从控节点超过 16 个时，用枚举逐节点记录状态的方式在 CAN 报文长度和处理效率上都会遇到瓶颈。本工程的 u32 offCell（32路位图）和 u16（16 路温度通道）的设计在8 字节 CAN帧内就能完整传输一个从控的全通道异常状态，这是面向实际通信约束的务实选择。

  1. nowState 与 filState 的分离是测试友好性的关键

在单元测试或台架联调阶段，可以直接调用 DiagSetErrFilStateAPI() 强制注入一个稳定故障状态（仅接受eErr_A 或eErr_B），绕过滤波延迟，快速验证保护动作是否正确触发。这种设计让”故障注入测试”不需要修改任何业务代码，只需通过开放的API 接口完成，大大降低了测试成本。

  1. 恢复次数限制是防止系统被绕过的安全机制

t_FilPara.maxTimes（最大可恢复次数）为 0时表示”无限次允许恢复”，设为 255则意味着”触发后永不自动恢复，必须人工复位”。对于继电器粘连这类硬件层面的故障，配置为 255是正确的——粘连在软件层面无法自愈，不应允许系统自动复位后继续运行，否则会掩盖真实的硬件问题。
