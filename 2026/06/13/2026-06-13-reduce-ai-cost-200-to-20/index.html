我是怎么把每月200刀省到20的

上个月我把Claude升级到Max版本，$200一个月。

第五天，额度没了。

打开使用记录一看，好家伙——一个下午调研47篇论文，一口气吃掉一周10%的额度。这种会话一周跑两三回，额度跟流水一样哗哗就没了。

我肉疼啊。

后来想明白了——问题不在Claude本身，而在于我一直让它干一件它根本不该干的事：当全文检索引擎用。

你往对话框里塞50k字符的日志，问一个问题，日志就被算一次input token。再问一个，好，命中了prompt cache，单价变成1/10了，但主会话还是会随轮次缓慢累加。更坑的是cache只有1小时TTL，间隔久了就得重来一次全价。

这感觉就像——每次问律师问题，都得先让他把你50页合同朗读一遍。

Claude Code最擅长的是推理、编排、写代码。读原始语料这种脏活累活，干嘛不让别的工具干？

顺着这个思路，我找到了NotebookLM。

一、NotebookLM到底香在哪

先说说我怎么被朋友安利的。

她写论文，reading list有60多篇。以前在PDF里Ctrl-F找半天，现在直接把全部论文丢进一个notebook，问一句“谁支持X观点、谁反对、分歧在哪”，答案直接带[1][2][3]引用甩过来，点一下就跳回原文。

她说一周能省十几个小时。

我本来不信，试了一周——好吧，我上瘾了。

这玩意儿牛在哪：

  • 免费档50个源，Pro档300个
  • 处理能力不要钱：上传、索引、生成、对话——全走Google的算力
  • 能直接把notebook变成音频播客：通勤的时候听，资料被两个AI用我没想过的角度讲一遍，经常能听出新东西
  • 格式从来不是问题：PDF、网页、YouTube字幕、Google Docs、图片OCR、音频转写——都能当源

就上面这些，NotebookLM已经是个很强的独立工具了。如果你需求就是“坐下来问几个问题”，用到这儿就够了。

但我用了两周，发现它有两个卡死我的点：

第一，tab切到崩溃。 调研一个课题：问一个问题 → 得到答案 → 点引用跳原文 → 读完 → 回notebook复制 → 切到Claude消费 → 做完实验 → 发现少一篇资料 → 切到Google搜 → 切到下载 → 切回notebook加源 → 继续问……一下午切200次tab。

第二，跟本地工具是两个世界。 我排查线上事故时把日志灌进notebook能查了，但我还要同时在终端grep配置、看k8s events、起pod复现——网页不能帮我跑任何本地命令。每次都是“网页看完 → 手动敲一遍 → 再切回网页”。

NotebookLM网页把自己定位成终点——你问它，它答你，结束。

但我需要它成为流水线上的一环——被调度、被批处理、输出能流到下一步。

怎么办？

把Claude套上去。

二、套上Claude，串起来用

核心思路特别简单：让NotebookLM当老师，让Claude当助手。

老师（NotebookLM）是只读的咨询台：你一次性把47篇论文灌进去就不用管了，它就在那儿待着等你问。不用回灌、不用喂新笔记——论文本身的观点已经足够撑起所有查询。

助手（Claude）负责写代码、跑脚本、整理结果、编排工具。不懂就去问老师，拿到答案继续干活。

你是课题负责人，只在关键决策点介入。

这就是省钱的全部秘密：

  • RAG vs 塞context是两种不同的成本模型。 把50k字符日志塞进Claude对话，这坨数据就被算进input token。每问一次就要被“看”一次，成本随语料大小线性涨。走RAG则是NotebookLM内部用向量检索命中相关片段，Claude只看到几百字的蒸馏答案，成本近乎常数。

  • prompt cache有1h TTL，研究场景命中率很低。 很多人以为开了cache就万事大吉。实际Anthropic的prompt cache默认只存1小时，超时就自动失效；你思考几分钟、切去做别的、或者开新session，下一次调用就得按全价重写一次cache_creation。研究性会话恰好是“问一下、想一会、再问一下”的节奏，命中率惨不忍睹。这是账单暴涨的真正原因。

  • 基于事实输出会更有效。 NotebookLM的答案被约束在你上传的源里，每句话带[1][2]引用，点回原文。不会胡编。Claude拿这种答案做决策，不用反复让它“再确认一下”，省下的是更难量化的时间成本。

简单说：真正省Claude token的办法不是开cache，是让重数据一开始就不进Claude。

三、10分钟装好，开干

Google官方没提供NotebookLM client，但有个第三方工具可以用——@icebear0828写的notebooklm-client。

GitHub搜一下就有了。

装完之后，在对话里说“查一下那个notebook里X的部分”，Claude会自动调——不用每次解释语法。

具体安装页面上有，10分钟能搞定。

四、实测：到底省多少钱

下面这组数不是模拟的，是真实一轮研究会话，从Claude Code的session log里扒出来的。

测试设置：

  • 语料：47篇图像+LiDAR SLAM相关论文，一次性灌进同一个NotebookLM notebook
  • 模型：Claude Opus 4.7
  • 轮数：5轮深度问答
  • 方式：Claude Code里正常对话，每轮助手自己调/notecraft chat去问老师

实测结果：

5轮贵档合计$0.55，平均每轮约$0.11。

关键数字：cache_creation只有17,379。

47篇论文的原文一个字都没进Claude的cache_creation——这就是省钱的全部秘密。

对比：如果把47篇论文直接塞prompt

47篇论文实测38.4万words≈50万token。

传统做法分两种场景，最公平的是“单session多轮对话”——同一研究会话里首轮建cache、后续复用，是传统做法的最优情况。

即使这样，5轮问答贵档差出17倍（$9.59 vs $0.55）。跨session场景只会更惨——86倍。

语料再翻倍（100篇、200篇），差距继续线性拉开。传统做法cache_creation随论文数线性涨，本文做法基本不变。

用Opus跑研究的人：200次研究会话一年就是小两千刀的差距——光“论文不进Claude”这一个动作，一年省下的够再升一次Max。

代价是什么？慢3倍。

  • 建notebook + 加一个源：10-15秒
  • NotebookLM chat一次：16-48秒（中位约45秒）
  • Claude Opus单问：20-35秒

如果你在乎响应速度超过账单，这套不适合你。

五、几个能直接抄的工作流

1. 学者/学生工作流

reading list是天然的知识边界。

一学期几十篇论文，同一批PDF要反复查几十次。以前Ctrl-F翻到眼花，问ChatGPT又怕它瞎编不带引用。

语料配方（灌一次用一学期）：

  • 20-50篇课题相关论文PDF
  • 课程大纲、讲座字幕
  • 导师邮件、自己的章节草稿、读书笔记

老师能回答的杀手问题：

  • “哪两篇结论互相冲突，冲突在哪一层假设？”
  • “X方法在这个语料里出现过几次，各自怎么用的？”
  • “A论文的公式3和B论文的公式7实际等价吗？”

Claude在链路里干嘛： 按课题推进——问老师拿概念/公式 → 写代码复现 → 跑实验 → 整理笔记。论文原文一次都不进Claude会话。

2. 打新股/读招股书工作流

一本招股书300-600页，打新窗口就三天。靠人读完再决策根本来不及。

港股/A股打新节奏快，新股上市前只有聆讯后资料集（港股）或招股意向书（A股）能看，文档动辄500页+。一家看完至少4小时，一周5-8家新股，根本来不及。

而且招股书最有信息量的不是“公司怎么夸自己”，是藏在风险因素、关联交易、历次融资估值里的那些红旗。这部分人眼最容易漏。

语料配方（一家公司一个notebook）：

  • 招股书全本——必灌，最核心
  • 基石投资者披露表
  • 同行可比公司最新财报
  • 保荐人/承销商研报
  • 管理层过往访谈、公司过去融资轮估值表
  • 行业监管政策文件

老师能回答的8个核心问题：

  1. 核心产品/业务是什么？近3年收入结构变化如何？客户集中度？
  2. 这家和同行在毛利率、营收增速、研发占比上的差异在哪？
  3. 基石投资者是谁、认购金额、锁定期？有没有知名产业资本？
  4. 募资用途按比例拆，哪部分最大？控股股东持股稀释到多少？
  5. 风险因素里，哪些是行业共性，哪些是公司特有？
  6. 过往融资估值：上轮估值/IPO估值的跳涨倍数？上轮投资人锁定期多久？
  7. 历史财务有没有一次性收益把利润做高的痕迹？经营现金流和净利润是否匹配？
  8. 关联交易占营收比多少？前五大客户里有没有关联方？

Claude在链路里干嘛： 批处理。把一周打新池里8家的一页汇总成一张markdown决策表，15分钟扫完下单。

一周打新池5-8家，全程招股书不进Claude对话——单本150-250k token，5家就是100万token的语料量，传统做法光这一项一周就能烧掉$50+。走本文做法$2以内。

3. 个人知识库工作流

给自己建“第二大脑”。

Obsidian搜索只认关键词，答不出“我对X的看法这三年变过没”。所有笔记都是自己写的，合规上没顾虑，但体量散、格式杂，本地没工具能跨文件做语义检索。

语料配方：

  • Obsidian/Notion全量导出
  • Kindle高亮、Readwise剪藏
  • 工作日记、会议纪要、复盘文档

老师能回答的杀手问题：

  • “我这三年对’专注力’写过什么？观点变了吗？”
  • “《原则》和《思考快与慢》对认知偏差的说法，哪里重叠哪里冲突？”
  • “过去一个月所有会议纪要里，X项目各人的态度分别是什么？”

三个工作流的共同点： 反复查、跨文档、私有边界——占任何一条，建库15秒成本一周内摊平。

最后说几句

有几个事值得注意：

  • storage_state.json里面是你Google的活session，注意保管
  • notebooklm-client是逆向NotebookLM内部协议做出来的，Google不认账，随时可能改后端让你的命令突然报错

这套东西的核心就一句话——分工：

  • NotebookLM当老师：答领域知识，带引用，不乱外推
  • Claude当助手：编排工具、写代码、整理结果，不懂就问老师
  • 你当课题负责人：只在关键决策点介入

我用了一个月，省下来的钱够再吃好几顿大餐。

但更重要的是，调研十几篇论文不再让我心疼额度了——这种“不用算账”的爽，比省钱本身还上瘾。
