Building a 200ms Grid Frequency Regulation Path in a Battery Storage System

Author: KuangYue
Date: 2026-06-01
Project: Solaris Battery Energy Storage Management System (ILC-A2415)

1. Why 200ms?

Grid frequency is the heartbeat of a power system. In China, the nominal frequency is 50Hz with a tolerance of ±0.2Hz. When frequency drops, generation is falling behind load — you need to inject active power fast. When it rises, the opposite. Traditional frequency regulation relies on thermal plant governors, which respond in seconds, sometimes minutes.

Lithium battery storage systems can respond in milliseconds. That’s the whole value proposition for grid frequency regulation (GridSvc) markets. The Chinese National Energy Administration and regional grid operators have codified this into a hard requirement: active power response within 200ms of receiving a dispatch command.

That 200ms is the constraint everything in this feature had to be designed around.

Our existing control path looked like this:

Under normal load, a full round-trip through that chain takes 500ms to 1 second. That’s not going to work.

The task: carve out a dedicated fast path for frequency regulation without touching the existing control architecture.

2. Understanding the Data Flow First

Before writing a single line of code, I spent time mapping out what we actually had. The SEMS is a layered system:

A few things worth noting about this architecture:

  • ModbusTCP2 Reporter is the entry point for the upper EMS. It speaks Modbus TCP, parses frames, and extracts active power commands.
  • SEMS CCU Sampler is the brain. It runs the EMS algorithm and decides how to distribute power across slave nodes.
  • SlaveEMS Reporter is the hands. It translates control commands into actual PCS hardware actions.
  • Inter-process communication uses System V semaphores — a standard lightweight synchronization primitive in embedded Linux.

We already had a fast path called QickAnt (for anti-reverse-flow and overload protection) that proved the “semaphore trigger + direct UDP” pattern works. GridSvc is essentially a second dedicated channel built on the same skeleton.

3. The Core Design: Two Independent Channels + Blocking Mode

3.1 Why a Separate Channel Instead of Reusing QickAnt?

The existing QickAnt channel uses UDP ports 9989/10005. The obvious question is: why not just reuse it?

Two reasons.

Semantic isolation. QickAnt is triggered by the 485 meter sampler (SEM_CCU_485_PATH). Its trigger timing has nothing to do with frequency regulation dispatch commands. Mixing them would couple two completely different control flows, making both harder to debug and maintain independently.

Different state machine behavior. When a slave receives a GridSvc command, it needs to execute a specific state transition using sem_binary_post rather than sem_post_op. That’s a meaningful behavioral difference. A dedicated channel lets the slave side handle them differently without conditional logic scattered everywhere.

The decision: a fully independent GridSvc channel — its own UDP port pair (9990/10006), its own semaphore (SEM_GRID_SERV), its own threads.

This is reflected cleanly in protocol_model.h:

The enum makes the communication topology self-documenting. When you see PROTOCOL_UDP_GridSvc anywhere in the codebase, you know exactly which physical channel and which use case it belongs to.

3.2 Blocking Mode for Modbus TCP: Eliminating Scheduler Jitter

The original ModbusTCP2 Reporter used a non-blocking multiplexed thread (_ThreadEntry_ModbusComm_NonBlock) based on select(). That’s fine for throughput — you can handle multiple clients concurrently. But under load, select() wakeup latency can drift by tens of milliseconds. For frequency regulation, that’s unacceptable.

What we need is deterministic latency, not high throughput. The upper EMS for frequency regulation is a single client sending a single command. So we added a blocking-mode thread:

Three details here that matter:

The accept timeout. Without it, accept() blocks forever and the thread can never exit cleanly. One second is a reasonable tradeoff — the system shuts down within a second of requesting it.

Forcing the socket to blocking mode. After accept(), we strip O_NONBLOCK from the new socket with fcntl. This means modbus_receive() blocks directly on the socket with zero polling overhead. The thread wakes up exactly when data arrives, not on a timer.

One connection at a time. This is the tradeoff of blocking mode. For frequency regulation, the upper EMS is a single dedicated connection. The constraint is acceptable.

3.3 Binary Semaphore vs. Counting Semaphore

GridSvc uses sem_binary_post. QickAnt uses sem_post_op. This difference is intentional.

A counting semaphore accumulates. If the upper EMS sends three commands in quick succession before the CCU Sampler processes the first one, the semaphore count reaches 3, and the sampler will execute EMS calculations three times in a row — processing stale commands that have already been superseded.

A binary semaphore saturates at 1. No matter how many times you post it, the value stays at 1. If the sampler hasn’t processed the previous command yet, the new one simply overwrites it. For frequency regulation, we care about the latest power setpoint, not a queue of historical ones.

The code in handle_frame.c reinforces this with an additional guard:

The semaphore only fires when the power value actually changes. If the upper EMS keeps sending the same setpoint (which happens in practice), we skip the entire downstream chain. Clean and cheap.

4. Real-Time Thread Scheduling: SCHED_RR and Priority 98

Standard Linux threads run under SCHED_OTHER, managed by the CFS scheduler. CFS is great for fairness and throughput. It is not great for latency guarantees. Under load, a SCHED_OTHER thread can sit in the run queue for tens of milliseconds before getting scheduled.

Every thread in the GridSvc fast path uses real-time scheduling:

SCHED_RR is round-robin real-time scheduling. A real-time thread at priority 98 will preempt any non-real-time thread and any real-time thread with lower priority. Linux real-time priorities run from 1 to 99; 98 is near the top.

PTHREAD_EXPLICIT_SCHED is the one that trips people up. The default is PTHREAD_INHERIT_SCHED, which means the new thread inherits the scheduling attributes of the thread that created it — completely ignoring whatever you set with pthread_attr_setschedparam. The comment in the code says it plainly: “this line is required for the priority to take effect.” It’s easy to miss and hard to diagnose when you do.

The same configuration applies to the blocking Modbus thread in modbustcp2_main.c. Every hop in the fast path is a real-time thread. That’s what makes the end-to-end latency deterministic.

5. Tracing a Command End-to-End

Here’s what actually happens when the upper EMS sends a new active power setpoint of 500kW:

Step 1 — Upper EMS writes Modbus register 3901 (t = 0ms)

The EMS client sends a Modbus TCP write to register address 3901 (MD_SIG_Ctrl_ActivePower).

Step 2 — ModbusTCP2 Reporter receives and parses (~5ms)

_ThreadEntry_ModbusComm_Block is blocked on modbus_receive(). It wakes immediately when the TCP data arrives, parses the frame, extracts the power value, writes it to the database, and posts the binary semaphore SEM_GRID_SERV.

Step 3 — CCU Sampler wakes, runs EMS algorithm (~20ms)

S_Thread_ServerUDP_GridSvc is blocked on sem_binary_wait(). The semaphore post wakes it immediately. It calls Ems_Mgmt_Main_SEMS() to compute power distribution across all slave nodes, then sends UDP control frames:

Note that GridSvc uses emACDC_WorkStatu_REMS (the work state from the REMS layer), while QickAnt uses the fixed STATU_ACDC_INVERTER. This is a real semantic difference between the two channels, not just a naming convention.

Step 4 — UDP packet hits the slave (~1ms)

Send_6330UctlWorkPwrAuto sends the control frame over UDP port 9990 to the slave’s port 10006. UDP has no handshake overhead. On a local network, this is sub-millisecond.

Step 5 — SlaveEMS Reporter receives and processes (~5ms)

Step 6 — Slave CCU drives the PCS (~10ms + ~50ms PCS response)

The slave CCU wakes on the semaphore, translates the power command into PCS control signals, and the PCS begins ramping output.

Estimated end-to-end latency:

Stage Estimated latency
Modbus TCP receive + parse ~5ms
Semaphore wakeup + EMS calculation ~20ms
UDP transmission ~1ms
Slave receive + parse ~5ms
Slave CCU + semaphore ~10ms
PCS power ramp ~50ms
Total ~91ms

That’s well inside the 200ms requirement, with over 100ms of headroom.

7. Symmetric Design: Master and Slave Mirror Each Other

The architecture is symmetric by design. The master (CCU Sampler) has two UDP send channels; the slave (SlaveEMS Reporter) has two UDP receive channels.

The slave’s slave_ems.h adds the corresponding data structures:

Create_SlaveUDP_GridSvc creates the receive thread; S_Thread_SlaveUDP_GridSvc handles incoming frames. The slave’s receive thread runs at priority 2 — much lower than the master’s send thread at 98. That’s intentional: the master needs to react to the upper EMS and dispatch quickly; the slave’s receive side has more slack and shouldn’t starve other critical tasks.

9. Things That Bit Us

PTHREAD_EXPLICIT_SCHED is not optional

First test run: thread priority wasn’t taking effect, response times were still jittery. Root cause: missing pthread_attr_setinheritsched(&attr, PTHREAD_EXPLICIT_SCHED). The default PTHREAD_INHERIT_SCHED silently ignores everything you set with pthread_attr_setschedparam. The code comment calls this out explicitly now. It’s the kind of bug that’s invisible until you measure latency under load.

Binary semaphore semantics need careful implementation

sem_binary_post and sem_binary_wait are wrappers around System V semop. The “check if value is 0, then set to 1” logic needs to be atomic at the semaphore level, not just at the application level. If you implement this naively with two separate semop calls, you have a race condition. The wrapper handles this correctly, but it’s worth understanding why it has to.

The accept timeout value matters

The 1-second timeout on accept() is a deliberate choice. Too short and you’re burning CPU checking bExit constantly. Too long and shutdown takes forever. One second means the system is responsive to shutdown requests without wasting cycles. If your shutdown SLA is tighter, reduce it.

Float comparison in power change detection

FLOAT_NOT_EQUAL exists for a reason. Direct != comparison on floats is unreliable due to representation errors. In a power control context, if the same 500.0kW setpoint gets represented as 499.9999… and 500.0001… on two consecutive receives, a naive != would trigger an unnecessary semaphore post and EMS recalculation. The macro uses an epsilon comparison to avoid this.

10. What 200ms Actually Requires

The 200ms requirement sounds simple. In practice, hitting it reliably means getting several things right simultaneously:

Deterministic wakeup — real-time scheduling (SCHED_RR, priority 98) ensures threads aren’t sitting in the run queue behind lower-priority work. Without this, you might hit 200ms on average but blow it under load.

No polling — blocking sockets mean threads wake exactly when data arrives. Polling introduces a latency floor equal to the poll interval, plus jitter.

Latest-wins semantics — binary semaphores ensure the system always acts on the most recent command, not a backlog of stale ones. In frequency regulation, a 100ms-old power setpoint is already wrong.

Isolated channels — keeping GridSvc separate from QickAnt means a problem in one doesn’t affect the other. It also means you can tune, debug, and monitor them independently.

None of these are individually complicated. The challenge is recognizing that you need all of them together, and that skipping any one of them will show up as latency spikes at the worst possible moment — when the grid is actually stressed and frequency regulation matters most.
