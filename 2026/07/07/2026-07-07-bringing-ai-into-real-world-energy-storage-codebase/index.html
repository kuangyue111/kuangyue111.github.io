From Reactive Alarms to Predictive Insight: Bringing AI into a Real-World Energy Storage Codebase

I. Preface: Why I Wrote This Article

The project I work on is a distributed Energy Storage Management System (ESMS). The main controller runs on ARM-Linux, with over a dozen peripheral device types including BMS (Battery Management System), PCS (Power Conversion System), energy meters, thermal controllers, fire suppression units, PV inverters, and CCU (Battery Cluster Control Units). The system has hundreds of sampler and reporter modules, and the core data flow looks like this:

Over the past six months of maintaining this codebase, I’ve been persistently bothered by one fundamental problem: the alarm mechanism is, by nature, reactive.

A concrete example: System.sql registers over a thousand alarms, such as:

The trigger expression [$,0,2240] means “fire when status parameter 2240 == 1.” In other words, an alarm only fires after the cell has already over-voltaged, the bus has already over-voltaged, or insulation has already dropped below the threshold. This is textbook passive fault response — the fault is already a fait accompli before anyone is paged.

In energy storage stations at or above 5 MWh scale, a single battery thermal runaway event can cost millions. I want to evolve this system from passive alarming to proactive early warning — letting AI signal risk hours or even days before equipment actually fails.

This article documents my complete thinking on that evolution: how to classify fault modes, which models to use for time-series anomaly detection, how to predict remaining useful life, whether to run inference at the edge or in the cloud, and whether digital twins and reinforcement learning can play a role. All discussion is grounded in actual code and data structures from this project, not abstract algorithms.

III. Fault Mode Classification: FMEA as the Scaffold for AI

3.1 Why Start with FMEA

Many AI predictive maintenance discussions jump straight to LSTM or Transformers. I have made that mistake: an AI model built without FMEA (Failure Mode and Effects Analysis) as a foundation produces a stream of “anomalies” that operations staff have no idea how to act on.

The correct order is: enumerate fault modes clearly → map each mode to observable signals → then choose a model.

IV. Time-Series Anomaly Detection: From Isolation Forest to LSTM Autoencoder

4.1 Why Thresholds Alone Are Not Enough

The current alarm logic is single-parameter threshold comparisons like [$,0,2240]==1. Its fundamental weaknesses are:

  • Static: Once set, a threshold does not adapt to equipment aging or seasonal temperature shifts.
  • Point-in-time: Bus voltage OK, insulation OK, temperature OK — therefore the system is OK? Not necessarily. All three could be drifting in the wrong direction simultaneously without crossing their individual thresholds.
  • Memoryless: Alarm evaluation looks only at the current frame, not the trend over the past hour.

Time-series anomaly detection addresses exactly these three gaps.

4.2 Isolation Forest: The First Line of Defense

Isolation Forest is my recommended Phase 1 model, for these reasons:

  1. Unsupervised: No fault samples needed (fault samples in an energy storage station are extremely scarce — maybe a handful per month).
  2. High-dimensional tolerance: Can accept 20+ feature dimensions simultaneously.
  3. Cheap to train: Trains in minutes on a CPU.
  4. Edge-deployable: The model is a collection of decision trees — tens of KB — runnable on ARM.

Output is an anomaly score in [0, 1]. I add one more rule on top: only escalate if the score exceeds 0.7 for three consecutive windows, to suppress single-frame noise.

4.3 LSTM Autoencoder: Detecting Pattern Drift

Isolation Forest does not understand temporal ordering. “Temperature rise followed by overvoltage” and “overvoltage followed by temperature rise” look identical to it, but carry completely different physical meanings.

An LSTM Autoencoder fills this gap: use the past N steps to reconstruct the current step; a large reconstruction error means pattern drift.

Key design decisions:

  • Window length: 60 steps recommended (10 s sampling = 10 minutes), covering one mini charge/discharge cycle.
  • Normal sample selection: Exclude the 1-hour window before and after each alarm event from the training set. This step is mandatory — otherwise the model learns fault patterns into its definition of normal.
  • Regularization: Battery degradation is a slow process. Validation must be split by time (train on early data, validate on later data), not randomly — random splitting causes data leakage.

4.4 How the Two Models Work Together

On the reporting path, this project already has an MQTT channel. I will reuse the alarm path via Sdgs_RaiseAlert() in sdgs_msghandler.c, adding a new alertId class (e.g., 30001 = “AI Warning — Battery Inconsistency Degradation”), so AI warnings travel through the existing upstream channel. Operations staff need no new tools.

V. Remaining Useful Life (RUL) Prediction: Making Battery Replacement Proactive

5.1 RUL Has the Highest Economic Value

Anomaly detection avoids accidents. RUL prediction optimizes asset management.

A lithium battery pack has an operational lifetime of roughly 8–10 years. If you can predict 3–6 months in advance that “this cluster will hit its capacity fade threshold in month X,” the operator can:

  • Source spare parts in advance, avoiding spot-market premiums
  • Schedule downtime during low electricity-price windows
  • Adjust station-level operating strategy (reduce charge/discharge depth on heavily degraded clusters to extend their life)

5.2 Capacity Fade Modeling: Physics + Data Hybrid

Pure data-driven models (learning the SOH sequence directly with an LSTM) have a fatal problem: poor extrapolation. Battery degradation accelerates non-linearly at the “knee point” near end of life, and historical data may never have observed post-knee behavior.

I prefer a physics baseline + data residual correction approach.

Physics baseline (semi-empirical formula):

$$
\text{SOH}(t) = 1 - \alpha \cdot \sqrt{t} - \beta \cdot N_{\text{cycles}} \cdot \text{DOD}^{\gamma}
$$

Parameters α, β, γ are fit by least squares using fields already in this project’s historical data (A_BLOB_TYPE_DATASTORAGE.fBattTotalChrgKWH_D, fBattTotalDisChrgKWH_D, fBattChrgTimes_D, etc.).

Data correction layer: use GBDT (e.g., LightGBM) to fit the residual between actual SOH and the physics baseline. Input features: temperature exposure time, average DoD, average discharge C-rate, etc.

RUL output: extrapolate the model to SOH = 0.8 (the industry-standard EOL threshold) and find that timestamp.

VI. From “Alarm” to “Early Warning”: Engineering Value of N-Hour Lead Time

6.1 Lead Time Determines Value

The advance notice a warning provides directly determines its commercial value:

Different models map to different time horizons:

  • Isolation Forest / LSTM Autoencoder: 5 minutes – 1 hour
  • Trend extrapolation (Kalman / ARIMA): hours – 1 day
  • RUL model: weeks – months

The engineering approach I advocate is multi-timescale parallelism — feed the same data stream to models at three time horizons independently, each producing its own warning.

6.2 Wiring Early Warnings into the Existing System

This project already has a complete alarm pipeline:

Rather than building a separate AI alarm pipeline, I will reuse the existing one:

  1. Write AI inference results as virtual parameters. Register a deviceModel in SQL (e.g., device 999 = “AI Engine”) whose status parameters are the AI outputs: anomaly score, RUL, per-FMEA mode probability, etc.

  2. Alarm rules use the existing expression syntax. For example:

    [999,3,1001] means: AI device 999, status parameter 1001 (consistency anomaly score).

  3. Zero ops-side intrusion. The frontend, IEC104 reporting, MQTT publishing, and database history all connect automatically.

This is the engineering philosophy of leveraging what already exists — treat AI as a “reasoning virtual device” plugged into the existing framework.

VII. Model Deployment: Edge vs. Cloud

7.1 Decision Table

7.2 My Actual Choice: Tiered Deployment

The edge runs only deterministic, real-time models. The cloud runs models that need large data volumes, slower cadence, and cross-site learning. Raw data does not go to the cloud in full — both for bandwidth and compliance reasons.

7.3 Engineering Challenges for Edge Inference

The main controller in this project is an ARM platform (see code/src/mkenv_arm_st for the build script), typically a Cortex-A53/A72, 4 cores, 1–2 GB RAM. Running AI inference on it requires:

  • Prefer TFLite Micro or a compiled ONNX Runtime ARM build.
  • INT8 quantization is nearly mandatory — floating-point models run 3–5× slower on ARM.
  • Inference scheduling must avoid the real-time windows of the sampler threads. In sampler_loader.c, samplers run as periodic threads. AI inference should run in the gaps or in a lower-priority thread.

My preferred approach is a new reporter module: ai_inference_reporter. It fetches feature data from the DAI interface (dai_model.h), runs inference, and writes results back to the parameter store. This path has already been validated in my recent work — a reporter accessing the parameter store bidirectionally via DAI is feasible (though BLOB writes require VAR_VALUE, not PARAM_VAL_GET — a known pitfall covered in section IX).

VIII. Digital Twin + Reinforcement Learning: From Prediction to Autonomy

8.1 Digital Twin Is Not Science Fiction

“Digital twin” has become a buzzword, but in the energy storage context it has a very specific meaning: running a physics simulation model in the cloud that stays synchronized with the real station’s state.

  • Input: Battery equivalent circuit model parameters (R0, R1, C1, …) + real-time SOC and temperature
  • Output: Predicted SOC, temperature, and voltage for the next hour under any given control strategy

The core of the twin is an electro-thermal coupled battery model. Mature open-source implementations already exist (PyBaMM, Cantera, etc.). I will connect it to this project’s BMS data stream:

  1. Real-time data syncs to the twin every minute; the twin performs parameter identification (Kalman/UKF).
  2. Before any control strategy is dispatched, run it through the twin first to check for anomaly triggers.
  3. If the twin flags a trigger, veto the strategy; otherwise pass it through to real hardware.

This is predictive maintenance at the control level — not just issuing warnings, but intercepting bad control decisions before they execute.

8.2 Reinforcement Learning: Safe Sandbox for Policy Exploration

The other use of the digital twin is as a training environment for RL. Energy storage dispatch is itself a classic RL problem:

  • State: SOC, electricity price, temperature, load forecast, battery degradation coefficient
  • Action: Charge power, discharge power, whether to participate in frequency regulation
  • Reward: Revenue from electricity arbitrage − degradation cost − anomaly penalty

Training RL directly on a real station is impossible — a single wrong decision can be catastrophically expensive. But training in the twin first, then doing a grey rollout (run in low-risk periods first, collect deviation data, then refine the model) is far more viable.

IX. Pitfalls and Engineering Recommendations

Here are the real pitfalls I have personally hit that are directly relevant to this project:

1. All data timestamps must be UTC.
In sampler_system.c, some places use gmtime_r and others use localtime_r — mixed usage. When training AI models, timezone misalignment throws every time window off. My recommendation: persist features to the database in UTC; convert to local time only at the display layer.

2. The BLOB write interface signature is easy to get wrong.
When I was implementing strategy dispatch recently, I passed PARAM_VAL_GET as the buffer to pDAI->Set. In dai_loader.c this gets cast as *(VAR_VALUE*)pBuffer, so a wrong buffer type is silently treated as garbage. When writing AI outputs back to the parameter store, follow the pattern at web_req_handler.c:619-626 — declare VAR_VALUE varV; varV.blobValue = ... and pass &varV.

3. Alarm expressions do not automatically observe AI output.
The alarm engine in this project is triggered periodically. AI output parameters must be actively pushed (using Push_VarValue / Push_BlobValue); otherwise the alarm engine will never see a value change.

4. Always give AI a kill switch.
Add an AI_PredictiveMaint = 0/1 config entry in System.cfg. If the model starts misbehaving, it can be disabled within one second. Operations staff will not trust an AI that cannot be turned off.

5. Model version and data version must be coupled.
“Which model was trained on which time range of data” is a lifesaver when debugging production issues. Add an nModelVersion field to A_BLOB_TYPE_DATASTORAGE and persist it alongside every inference result.

X. Closing Thoughts

The current codebase of this energy storage management system is highly engineered — alarming, reporting, and protocol adaptation are all solid. But it is fundamentally reactive: wait for a fault, then handle it. AI is not here to replace any existing module; it is here to give the system a pair of eyes that can see ahead.

My expectations for this path:

  • Within 6 months: The edge anomaly detection model goes live, bringing fault modes like “cell inconsistency degradation” — which have no entry in the current alarm database — into the early warning system.
  • Within 1 year: The cloud RUL model goes live, providing operators with quarterly asset health reports.
  • Within 2 years: The digital twin + RL closed loop goes live, upgrading strategy dispatch from “passive execution” to “proactive optimization.”

This is not an AI project built to impress. It is an engineering evolution that makes an existing system more reliable.
